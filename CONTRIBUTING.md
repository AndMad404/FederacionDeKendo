# Contributing

## Commit messages

Manual commits must explain the problem, the intended outcome, and how the
change was verified. Use this structure:

```text
type(scope): observable outcome

Problem: why the change was needed
Intent: the behavior or result the change is meant to produce
Verification: commands or checks performed
```

Use one of these types: `fix`, `feat`, `refactor`, `docs`, `test`, `build`,
`ci`, or `chore`. Choose a short feature or concern for `scope`, such as
`gallery`, `affiliates`, `calendar`, `seo`, or `review`. Describe behavior or
an outcome in the subject instead of a filename or an activity such as
"update" or "touch".

Example:

```text
fix(gallery): replace mobile dots with swipe navigation

Problem: gallery dots consume space and do not match the mobile interaction.
Intent: make swipe navigation the only mobile gallery control.
Verification: pnpm run typecheck; pnpm run build
```

When a change has no executable check, record the reason explicitly:

```text
Verification: not run (documentation-only change)
```

This convention applies to commits written manually. Automated commits and
messages generated by Git, including merge and revert commits, are exempt.

To enable the repository template in a clone, run:

```bash
git config --local commit.template .gitmessage
```
